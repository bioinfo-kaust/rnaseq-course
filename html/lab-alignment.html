Lab 4: Read Alignment with STAR

In this lab, you will align the trimmed, subsampled paired-end reads from GSE136366 (a TDP-43 RNA-seq study in human cells) to the GRCh38 reference genome using STAR. Each sample has been subsampled to 1 million read pairs in Lab 2 to keep runtimes manageable during the workshop. You will then post-process and index the resulting BAM files, assess alignment quality, and visualize the alignments in IGV.

Using subsampled data in this lab

The input files (trimmed/${SRR}_trimmed_1.fastq.gz) were produced in Lab 3 by running fastp on the subsampled FASTQ files (raw_data/${SRR}_sub_1/2.fastq.gz, 1 M read pairs each) generated in Lab 2. With 1 M reads, STAR alignment completes in 2–5 minutes per sample rather than the 30–60 minutes needed for a full dataset. All commands are identical to a full-scale analysis — only the data size differs.

Learning Objectives
  • Build a STAR genome index from the GRCh38 FASTA and GTF annotation
  • Align paired-end reads to GRCh38 using STAR with appropriate parameters
  • Post-process BAM files: sort, index, and mark PCR duplicates with Picard
  • Assess alignment quality using samtools flagstat and the STAR alignment log
  • Visualize aligned reads at a gene locus using IGV

Why 32 GB RAM?

STAR loads the entire genome index into memory during alignment. The human GRCh38 index requires approximately 28–32 GB of RAM regardless of how many reads you are aligning — even with subsampled data. The read processing itself is fast (a few minutes for 1 M read pairs). If your interactive session request times out in the queue, submit alignment as a batch job (covered in Part 3).

Part 2: Build the STAR Genome Index

STAR requires a pre-built genome index that combines the reference FASTA and the gene annotation (GTF). This index is built once and reused for all samples. Building the human GRCh38 index takes approximately 1 hour and requires ~30 GB RAM, so it must be submitted as a SLURM batch job.

Tip: Check for a Pre-built Index First

We maintains a shared reference area. Before building your own index, check whether a STAR index for GRCh38 already exists:

ls /biocorelab/BIX/resources/genomes/Ensembl-release-113/GRCh38.p14/index/star/

If the directory exists and contains files like Genome, SA, and SAindex, you can use it directly in Part 3 by setting --genomeDir /biocorelab/BIX/resources/genomes/Ensembl-release-113/GRCh38.p14/index/star/ and skip the index-building step entirely.

Create the SLURM batch script for index generation

Save the following script as star_index.sh in your workshop directory:

cat > /ibex/user/$USER/workshop/star_index.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=star_index
#SBATCH --cpus-per-task=8
#SBATCH --mem=64G
#SBATCH --time=02:00:00
#SBATCH --output=logs/star_index_%j.log

module load star

mkdir -p references/star_index

STAR --runMode genomeGenerate \
    --genomeDir references/star_index \
    --genomeFastaFiles references/GRCh38_genome.fa \
    --sjdbGTFfile references/GRCh38_genes.gtf \
    --runThreadN 8 \
    --genomeSAindexNbases 14
EOF

Submit the index job and monitor progress

# Make sure you are in your workshop directory
cd /ibex/user/$USER/workshop

# Submit the batch job
sbatch star_index.sh

# Check that the job is running
squeue -u $USER

# Watch the log file as it progresses
tail -f logs/star_index_*.log
Key STAR Index Parameters
ParameterValueMeaning
--runMode genomeGenerateTells STAR to build an index rather than align reads
--genomeDirreferences/star_indexOutput directory for the index files
--genomeFastaFilesGRCh38_genome.faReference genome FASTA (can be multi-chromosome)
--sjdbGTFfileGRCh38_genes.gtfGene annotation used to add known splice junctions to the index
--genomeSAindexNbases14Suffix array index bases; 14 is appropriate for large genomes like human
--runThreadN8Number of CPU threads to use

Verify the index was built successfully

Part 3: Align Reads with STAR

With the genome index ready, you can now align all samples interactively. With subsampled data (1 M read pairs each), STAR finishes each sample in about 2–5 minutes, so a simple loop in your interactive session works well.

Run STAR alignment for all samples

cd /ibex/user/$USER/workshop

for srr in $(cat raw_data/SRR_Acc_List.txt); do
    echo "=== Aligning $srr ==="
    mkdir -p aligned/${srr}

    STAR --runThreadN 8 \
        --genomeDir references/star_index \
        --readFilesIn trimmed/${srr}_trimmed_1.fastq.gz trimmed/${srr}_trimmed_2.fastq.gz \
        --readFilesCommand zcat \
        --outSAMtype BAM SortedByCoordinate \
        --outSAMattributes NH HI AS NM MD \
        --outFileNamePrefix aligned/${srr}/ \
        --outSAMstrandField intronMotif \
        --alignSJoverhangMin 8 \
        --alignSJDBoverhangMin 1 \
        --quantMode GeneCounts \
        --runMode alignReads

    echo "Done: $srr"
done
Expected runtime

Each subsampled sample (1 M read pairs) takes roughly 2–5 minutes. The loop will process all samples sequentially. STAR prints progress to the terminal; a line like ... done mapping and the appearance of the Aligned.sortedByCoord.out.bam file confirm success for each sample.

Understand the key STAR alignment parameters

Part 4: BAM Post-processing

After alignment, BAM files need to be indexed for fast random access, and PCR duplicates should be flagged using Picard MarkDuplicates. These steps operate on the subsampled BAMs produced in Part 3 — the commands are identical to a full-scale workflow.

Index the BAM files with samtools

A BAM index (.bai file) allows tools like IGV and samtools to jump directly to any genomic region without reading the entire file.

cd /ibex/user/$USER/workshop

# Index the BAM file for every sample
for srr in $(cat raw_data/SRR_Acc_List.txt); do
    echo "Indexing $srr ..."
    samtools index aligned/${srr}/Aligned.sortedByCoord.out.bam
done

# Verify — each BAM should now have a matching .bai file
ls -lh aligned/$(head -1 raw_data/SRR_Acc_List.txt)/

Mark PCR duplicates with Picard

for srr in $(cat raw_data/SRR_Acc_List.txt); do
    echo "Marking duplicates for $srr ..."
    picard MarkDuplicates \
        I=aligned/${srr}/Aligned.sortedByCoord.out.bam \
        O=aligned/${srr}/marked_dup.bam \
        M=aligned/${srr}/dup_metrics.txt
    samtools index aligned/${srr}/marked_dup.bam
done
What is Duplicate Marking and Why Does It Matter?

During library preparation, PCR amplification can create multiple identical copies of the same original DNA molecule. These duplicates are not independent observations of gene expression — they are technical artifacts. Picard MarkDuplicates identifies read pairs that align to exactly the same position and flags them in the BAM file (it does not delete them by default). Downstream tools like GATK and many DEG pipelines can then ignore flagged duplicates.

Key output: dup_metrics.txt reports the duplication rate. For RNA-seq libraries, rates of 10–40% are common. Very high rates (>60%) may indicate poor library complexity. With subsampled data the ESTIMATED_LIBRARY_SIZE will appear smaller than a full-dataset run — this is expected.

Inspect the Picard duplication metrics

# View the duplication report for one sample
SRR=$(head -1 raw_data/SRR_Acc_List.txt)
cat aligned/${SRR}/dup_metrics.txt
Tip: Reading Picard MarkDuplicates Output

The key metric to check is PERCENT_DUPLICATION in the metrics file. Also note ESTIMATED_LIBRARY_SIZE — a low value suggests the library was sequenced to near-saturation. With subsampled data this estimate will be lower than in a full run; the duplication rate itself remains a meaningful quality indicator.

Part 5: Alignment Quality Control

Before proceeding to quantification, assess the quality of the alignments to make sure the data is suitable for downstream analysis. These steps work directly on the subsampled BAMs — with only 1 M read pairs per sample, flagstat and log inspection complete in seconds.

Run samtools flagstat on all samples

cd /ibex/user/$USER/workshop

# Print flagstat output for every sample
for srr in $(cat raw_data/SRR_Acc_List.txt); do
    echo "=== $srr ==="
    samtools flagstat aligned/${srr}/Aligned.sortedByCoord.out.bam
    echo ""
done
Key flagstat Fields to Check
FieldWhat to Look For
totalTotal reads in the BAM; with subsampled data expect ~2 M (1 M read pairs × 2 reads each)
mappedNumber and % of reads that aligned; aim for >80%
properly pairedBoth reads in a pair aligned concordantly; should be high (>90% of mapped)
singletonsReads whose mate did not map; a few percent is normal, high values may indicate quality issues

Inspect the STAR alignment summary log

# View the STAR summary log for the first sample
cat aligned/$(head -1 raw_data/SRR_Acc_List.txt)/Log.final.out

# Loop to print key stats for all samples
for srr in $(cat raw_data/SRR_Acc_List.txt); do
    echo "=== $srr ==="
    grep -E "Uniquely mapped|mapped to multiple|unmapped" aligned/${srr}/Log.final.out
    echo ""
done

Warning: What to Do If the Mapping Rate is Low

If uniquely mapped reads fall below 70%, investigate the following:

  • Wrong reference genome: Confirm you are using the correct species and genome build (GRCh38 for human).
  • Adapter contamination: Check FastQC reports from Lab 3. Run fastp again with more aggressive adapter trimming if needed.
  • rRNA contamination: A large fraction of unmapped reads may be ribosomal RNA not depleted during library preparation.
  • Incorrect library type: Check whether reads are stranded or unstranded; adjust --outSAMstrandField accordingly.
  • Short reads after trimming: If many reads are flagged as "too short", relax the minimum length threshold in fastp.
Unique vs. Multi-mapping Reads

Uniquely mapped reads align to exactly one location in the genome. They are the most reliable for quantification because their genomic origin is unambiguous.

Multi-mapping reads align equally well to two or more locations (e.g., reads from gene families or repeated elements). Different tools handle them differently: some discard them, some distribute them probabilistically. STAR outputs them in the BAM with an NH tag > 1. For standard differential expression analysis with tools like DESeq2, it is common to use only uniquely mapped reads.

Part 6: Visualize Alignments in IGV

The Integrative Genomics Viewer (IGV) lets you visually inspect read alignments at any genomic locus — essential for sanity-checking results and exploring specific genes of interest. Because these are subsampled BAMs (~1 M reads), coverage will be lower than a full dataset but gene structure, splice junctions, and strand orientation are still clearly visible.

What to Look for in IGV
  • Coverage track: The histogram at the top of the alignment track shows read depth per position. Exons should show peaks of coverage; introns should be nearly empty. With subsampled data, depth will be low (~5–30× for expressed genes) but the exon/intron contrast is still visible.
  • Splice junctions: Curved arcs connecting exons indicate reads that span splice junctions. You should see arcs connecting TARDBP exons.
  • Strand orientation: Read colors indicate which strand a read aligns to. For a stranded library, reads on each gene should be predominantly one color.
  • Mismatches: Colored bars within reads indicate mismatches to the reference. A few scattered mismatches are normal; clustering may indicate a variant or sequencing error.
Tip: Useful IGV Keyboard Shortcuts
  • + / - — zoom in and out
  • Ctrl+F — jump forward along the chromosome
  • Right-click on a read — see full alignment details, flags, and tags
  • Right-click on the track — change color scheme, squish/expand reads, sort by various criteria
